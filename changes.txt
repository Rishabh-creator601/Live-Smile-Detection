Changes Log
===========

File modified: Detect Smile.py

1. Added WebRTC STUN server configuration
   - Added an RTC_CONFIGURATION dictionary with Google's public STUN server
     (stun:stun.l.google.com:19302).
   - Passed it to webrtc_streamer() via the new rtc_configuration argument.
   - Why: After deployment the browser and server are on different networks
     behind NAT/firewalls. Without a STUN/TURN (ICE) server, WebRTC cannot
     negotiate a peer connection, which caused the error:
     "Connection is taking longer than expected. Check your network or ask
     the developer for STUN/TURN settings if the problem persists."
   - Note: STUN alone may fail on strict/symmetric NAT networks. If the error
     persists for some users, a TURN server (e.g. Open Relay, Twilio, or a
     self-hosted coturn) should be added to the iceServers list, with
     credentials stored in st.secrets rather than hardcoded.

1b. Added free public TURN servers (Open Relay / metered.ca) to RTC_CONFIGURATION
   - STUN alone was not enough on the cloud deployment, so TURN relay servers
     were added (ports 80, 443, and 443 over TCP) using the public
     openrelayproject credentials.
   - Why: Cloud deployments (e.g. Streamlit Community Cloud) usually require
     media to be relayed through a TURN server; STUN cannot relay traffic.
   - Caveat: These free public TURN credentials are for testing and may be
     rate-limited or unreliable. For production use a dedicated TURN provider
     such as Twilio Network Traversal Service (dynamic credentials via API)
     or a self-hosted coturn server, with credentials read from st.secrets.

3. Fixed OpenCV ImportError on deployment
   (ImportError: libgthread-2.0.so.0: cannot open shared object file)
   - requirements.txt: removed opencv-python==4.11.0.86 and kept only
     opencv-python-headless==4.11.0.86. Having both installed caused the
     GUI-dependent build to be used, which needs system GUI libraries not
     present on a headless server.
   - packages.txt: kept only libgl1. (libglib2.0-0 was briefly added as a
     safety net but REMOVED because the deployment's apt repo could not satisfy
     its dependencies — libffi7/libpcre3 not installable — which broke the
     build. It is not needed once opencv-python-headless is the only OpenCV.)
   - Why: The server has no display/GUI libraries. opencv-python-headless is
     the build intended for servers and avoids these missing-.so errors, so no
     extra system GUI libraries are required.

2. Fixed frame return bug in SmileDetector.recv()
   - Before: the frame was only returned when (frame_count % 5 != 0), so every
     5th frame returned None.
   - After: recv() now always returns the processed video frame.
   - Why: recv() must always return a frame; returning None every 5th frame
     caused visible video stutter. The skip logic was also incomplete since it
     never actually skipped the face-detection work, only dropped the output.

4. Reworked the app to follow the official streamlit-webrtc deployment pattern
   - Replaced the static RTC_CONFIGURATION dict with a get_ice_servers() helper:
       * If TWILIO_ACCOUNT_SID / TWILIO_AUTH_TOKEN are present in st.secrets, it
         fetches short-lived TURN credentials from Twilio's Network Traversal
         Service (client.tokens.create().ice_servers). This is the method the
         streamlit-webrtc docs recommend for deployed apps and is the reliable
         fix for the "Connection is taking longer than expected" error.
       * If those secrets are NOT set, it falls back to the free public
         STUN + Open Relay TURN servers (fine for testing, may be unreliable).
   - webrtc_streamer() now receives rtc_configuration={"iceServers": get_ice_servers()}.
   - Decorated load_detector() and load_model_file() with @st.cache_resource so
     the Haar cascade and Keras model load once instead of on every rerun.
   - Cleaned up unused imports (numpy) and the leftover "skip frames" comment;
     FPS is now stored on the processor instead of an unused local variable.
   - requirements.txt: added twilio==9.3.7 for the dynamic TURN credentials.

   How to enable Twilio TURN (recommended for production):
     1. Create a free Twilio account and copy the Account SID and Auth Token.
     2. In the deployment's secrets (Streamlit Cloud: App settings -> Secrets),
        add:
            TWILIO_ACCOUNT_SID = "ACxxxxxxxx..."
            TWILIO_AUTH_TOKEN  = "your_auth_token"
     3. Redeploy. Without these the app still runs using the free fallback.

5. Reverted change #4 (per request: "undo the code to web rtc")
   - Detect Smile.py: removed the Twilio get_ice_servers() helper and the
     @st.cache_resource decorators. Restored the plain streamlit-webrtc setup
     with a static RTC_CONFIGURATION (Google STUN + free Open Relay TURN), as
     in change #1/#1b.
   - requirements.txt: removed twilio==9.3.7 (no longer needed).
   - Net effect: the app uses streamlit-webrtc with the static STUN/TURN
     config; no Twilio dependency or secrets required.
